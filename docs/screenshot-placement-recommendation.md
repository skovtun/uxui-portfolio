# Рекомендация по размещению скриншотов приложений в кейсе Empower

## Контекст
Вопрос: где лучше разместить скриншоты приложений, созданных с использованием дизайн-системы — в начале кейса или в конце?

## Анализ текущей структуры

### Текущее размещение (начало кейса):
1. **Corporate Site UI Kit** — показывает систему компонентов
2. **Core Components** — базовые компоненты
3. **Stats Block** — метрики (150+ компонентов, 18 виджетов, 12+ проектов)
4. **Corporate Website** (desktop + mobile) — результат
5. **Financial Dashboard** — результат

### Структура кейса:
- **Hero Section** (изображение проекта)
- **Overview**
- **Context & Problem**
- **Solution**
- **Result First** (UI Kit, Components, Stats, Website, Dashboard) ← **ЗДЕСЬ СЕЙЧАС**
- **Approach**
- **Design System Details** (архитектура, токены, компоненты, паттерны)
- **Impact**
- **Reflection**

## Рекомендация: **ОСТАВИТЬ В НАЧАЛЕ** ✅

### Аргументы ЗА размещение в начале:

#### 1. **Hiring Manager Narrative (FAANG-подход)**
- **Показываем результат сразу** — это то, что хотят видеть рекрутеры
- **"Show, don't tell"** — визуальное доказательство работы важнее описания процесса
- **Первое впечатление** — скриншоты приложений создают сильное первое впечатление о масштабе и качестве работы

#### 2. **Психология восприятия**
- **Внимание рекрутера ограничено** — первые 30 секунд критичны
- **Визуальная якорь** — скриншоты создают "якорь" для оценки всего кейса
- **Эмоциональная связь** — реальные приложения вызывают больше доверия, чем схемы компонентов

#### 3. **Структура "Result → Process → Impact"**
- **Результат первым** — показывает, что было достигнуто
- **Процесс вторым** — объясняет, как это было сделано
- **Impact последним** — подтверждает ценность

#### 4. **Оптимизация для сканирования**
- Рекрутеры **сканируют**, а не читают детально
- Скриншоты приложений легко сканируются и дают быструю оценку
- Технические детали (компоненты, токены) важны, но вторичны

### Аргументы ПРОТИВ размещения в конце:

#### 1. **Антиклимакс**
- После детального описания системы скриншоты могут показаться "очевидными"
- Риск, что рекрутер не дойдет до конца

#### 2. **Логическая последовательность**
- Может показаться логичнее: "вот система → вот что с ней сделали"
- Но это работает только для технических ролей, не для hiring managers

## Рекомендуемая структура (текущая — оптимальна):

```
1. Hero Image
2. Overview
3. Context & Problem
4. Solution
5. **RESULTS FIRST** ← Скриншоты приложений здесь
   - Corporate Site UI Kit
   - Core Components
   - Stats (150+ components, etc.)
   - Corporate Website (desktop + mobile)
   - Financial Dashboard
6. Approach
7. Design System Details (глубокие технические детали)
8. Impact
9. Reflection
```

## Дополнительные рекомендации:

### 1. **Улучшить визуальную иерархию результатов**
- Сделать скриншоты приложений более заметными (больше размер, лучше качество)
- Добавить краткие подписи, подчеркивающие масштаб использования системы

### 2. **Связь результатов с метриками**
- Stats Block (150+ компонентов, 12+ проектов) должен быть **сразу после** скриншотов
- Это создает связь: "вот приложения → вот масштаб"

### 3. **Оптимизация для разных аудиторий**
- **Hiring Managers** — видят результаты сразу (сканируют начало)
- **Design Leads** — читают детали системы (читают до конца)
- **Оба** получают то, что нужно

## Вывод

**ОСТАВИТЬ СКРИНШОТЫ В НАЧАЛЕ** — это правильное решение для FAANG-level кейса.

Текущая структура оптимальна, потому что:
- ✅ Показывает результат первым (hiring manager narrative)
- ✅ Создает сильное первое впечатление
- ✅ Соответствует ожиданиям рекрутеров
- ✅ Балансирует визуальное и текстовое содержание

**Единственное улучшение**: убедиться, что скриншоты приложений визуально выделяются и имеют качественные подписи, подчеркивающие масштаб использования дизайн-системы.
